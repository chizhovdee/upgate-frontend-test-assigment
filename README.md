# Frontend Test Assignment

Тестовое задание для frontend-разработчика. Нужно реализовать **SPA-приложение**
интернет-магазина: каталог товаров → корзина → форма оплаты → Loader (ожидание).

**Стек: React + react-router**, сборщик — **Vite** (SPA с клиентским
роутингом). Бэкенд уже готов (`server.js`), его менять не нужно — только
использовать.

## Запуск бэкенда

```bash
npm run server
```

Сервер поднимется на `http://localhost:3000`.

### API

| Метод | Эндпоинт    | Описание                                               |
| ----- | ----------- | ------------------------------------------------------ |
| GET   | `/products` | Возвращает массив товаров (≈1000 шт).                  |
| GET   | `/state`    | Long-polling статуса оплаты (см. ниже).                |
| POST  | `/submit`   | Принимает данные карты, возвращает `200 { ok: true }`. |

**`/products`** — массив товаров. Структура одного объекта (см. `data.json`):

```json
{
  "id": 1,
  "title": "Fjallraven - Foldsack No. 1 Backpack",
  "price": 109.95,
  "description": "Your perfect pack for everyday use...",
  "category": "men's clothing",
  "image": "https://fakestoreapi.com/img/....png",
  "rating": { "rate": 3.9, "count": 120 }
}
```

**`/state`** — возвращает `200 { action: "WAIT" }` случайное число раз (≥ 3),
после чего `200 { action: "SUCCESS" }`. Счётчик и случайный таргет сбрасываются
при каждом `POST /submit`.

**Нестабильная сеть (общая логика).** Любой запрос к серверу может случайно
вернуть `503` (сетевая ошибка). Это нужно корректно обрабатывать и повторять
запрос.

**`/submit`** — тело запроса:

```json
{
  "cardNumber": "string",
  "email": "string",
  "cvv": "string",
  "expire": "string",
  "productIds": [1, 2, 3]
}
```

Сервер **валидирует** объект по паттернам ниже. Невалидный объект приходить
не должен — фронт обязан отправлять только корректные данные. Если валидация
не пройдена, сервер вернёт `400 { error, fields }`.

| Поле         | Паттерн                      | Описание                   |
| ------------ | ---------------------------- | -------------------------- |
| `cardNumber` | `^\d{16}$`                   | 16 цифр (пробелы обрезать) |
| `email`      | `^[^\s@]+@[^\s@]+\.[^\s@]+$` | корректный email           |
| `cvv`        | `^\d{3,4}$`                  | 3 или 4 цифры              |
| `expire`     | `^(0[1-9]\|1[0-2])\/\d{2}$`  | формат `MM/YY`             |
| `productIds` | массив целых чисел            | id товаров, существующие в каталоге |

## Что нужно сделать

### 1. Страница каталога товаров

- Реализовать **свой компонент `DataGrid`**. На вход принимает **только** два
  пропса: `data` и `columns`. Никаких сторонних grid-библиотек.
- Можно выбрать несколько товаров (multi-select) и добавить их в корзину.
- Корзина хранится в **`localStorage`** (переживает перезагрузку страницы).
  Именно `id` товаров из корзины уходят в `productIds` при оплате.
- Кнопка **Checkout** — переход на страницу формы оплаты.

### 2. Страница формы оплаты (Checkout)

Поля ввода с **валидацией** (паттерны — см. раздел `/submit` выше):

- Механизм валидации и подсветки полей должен быть **своим** (без библиотек валидации).
- `cardNumber` — номер карты
- `email` — email
- `cvv` — CVV
- `expire` — срок действия карты

**Поведение валидации:**

- Поле подсвечивается (краснеет) при невалидных данных **только если** оно уже
  было «тронуто» (`touched`) **или** была нажата кнопка submit.
- Пока поле не тронуто и форма не отправлялась — ошибки не показываем.
- Кнопка submit отправляет запрос только при полностью валидной форме; на бэк
  всегда должен приходить валидный объект.
- Если поле было изменено и пользователь хочет закрыть вкладку, нужно предупреждать его что остались не сохраненные данные

По нажатию на кнопку отправки — `POST /submit` с заполненными полями.
Пример тела запроса:

```json
{
  "cardNumber": "4242424242424242",
  "email": "some@mail.com",
  "cvv": "302",
  "expire": "12/26",
  "productIds": [12, 25]
}
```

После успешного `/submit` — **очистить корзину** и перейти на страницу Loader.

### 3. Страница Loader

- Шлёт long-polling на `GET /state`.
- При `action: "WAIT"` — продолжать опрашивать `/state`.
- При `action: "SUCCESS"` — показать сообщение «Покупка была успешной».

## Общие требования

- При нажатии кнопки **«Назад»** в браузере клиента всегда должно
  возвращать на страницу с товарами (каталог).
- Чистый, читаемый код и структура проекта приветствуются.
- Можно использовать любой CSS-фреймворк или писать стили самому. Компонент
  `DataGrid` должен быть собственной реализацией — без сторонних grid/table-библиотек.
- Форма оплаты и страница Loader должны загружаться быстро. Подумать над
  оптимизацией загрузки.
- Для выполнения задания нужно сделать fork проекта и прислать ссылку на GitHub-репозиторий для проверки.
